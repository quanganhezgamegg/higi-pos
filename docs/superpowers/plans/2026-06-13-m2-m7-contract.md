## 1. Complete CREATE TABLE SQL (master spec §5)

Conventions enforced everywhere:
- `id INTEGER PRIMARY KEY` (SQLite rowid alias).
- **Money = `INTEGER` VND**, no decimals, `NOT NULL DEFAULT 0` unless nullable by domain (e.g. `tendered`).
- Booleans = `INTEGER NOT NULL DEFAULT 1` (active) / `DEFAULT 0`, read in Rust via `row.get::<_, i64>(..)? != 0`.
- `sort_order INTEGER NOT NULL DEFAULT 0`; lists ordered `ORDER BY sort_order, id`.
- `is_active INTEGER NOT NULL DEFAULT 1`; soft-disable instead of deleting referenced rows.
- Timestamps = `TEXT` UTC ISO-8601 (e.g. `2026-06-13T08:30:00Z`). Rust writes them; default via `strftime` only where noted.
- Enums stored as `TEXT` with a `CHECK(... IN (...))` constraint.
- Snapshot columns (name/price) on `order_items` / `order_item_toppings` / `order_discounts` so historical bills/reports never change when the menu changes.

### M2 — `0003_tables.sql`

```sql
CREATE TABLE areas (
  id         INTEGER PRIMARY KEY,
  name       TEXT NOT NULL,
  sort_order INTEGER NOT NULL DEFAULT 0,
  is_active  INTEGER NOT NULL DEFAULT 1
);

CREATE TABLE tables (
  id         INTEGER PRIMARY KEY,
  area_id    INTEGER NOT NULL REFERENCES areas(id),
  name       TEXT NOT NULL,
  seats      INTEGER,
  sort_order INTEGER NOT NULL DEFAULT 0,
  is_active  INTEGER NOT NULL DEFAULT 1
);
CREATE INDEX idx_tables_area ON tables(area_id);
```

> `tables` is a reserved-ish word in some engines but is valid in SQLite; keep the name `tables` to match the spec. Deleting an `area` that still has `tables` is blocked in the repo layer (mirror M1's `delete_category` guard).

### M3 — `0004_orders.sql`

```sql
CREATE TABLE orders (
  id             INTEGER PRIMARY KEY,
  code           TEXT NOT NULL UNIQUE,
  order_type     TEXT NOT NULL CHECK (order_type IN ('DINE_IN','TAKEAWAY')),
  table_id       INTEGER REFERENCES tables(id),
  shift_id       INTEGER,                 -- NULLABLE, NO hard FK (resolved in §2); set by M5 logic
  status         TEXT NOT NULL DEFAULT 'OPEN' CHECK (status IN ('OPEN','PAID','CANCELLED')),
  subtotal       INTEGER NOT NULL DEFAULT 0,
  discount_total INTEGER NOT NULL DEFAULT 0,
  total          INTEGER NOT NULL DEFAULT 0,
  note           TEXT,
  created_at     TEXT NOT NULL,
  paid_at        TEXT
);
CREATE INDEX idx_orders_status   ON orders(status);
CREATE INDEX idx_orders_table    ON orders(table_id);
CREATE INDEX idx_orders_shift    ON orders(shift_id);
-- At most ONE OPEN order per table (table "đang phục vụ" suy ra từ đây).
CREATE UNIQUE INDEX uq_orders_open_table
  ON orders(table_id) WHERE status = 'OPEN' AND table_id IS NOT NULL;

CREATE TABLE order_items (
  id            INTEGER PRIMARY KEY,
  order_id      INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id    INTEGER REFERENCES products(id),   -- nullable: product may be deleted later
  product_name  TEXT NOT NULL,                     -- snapshot
  size_name     TEXT,                              -- snapshot (nullable)
  unit_price    INTEGER NOT NULL,                  -- snapshot, ALREADY includes size price_delta
  quantity      INTEGER NOT NULL DEFAULT 1 CHECK (quantity > 0),
  sugar_level   TEXT,                              -- snapshot label, e.g. '50%'
  ice_level     TEXT,                              -- snapshot label, e.g. 'Ít'
  line_note     TEXT,
  line_discount INTEGER NOT NULL DEFAULT 0,        -- item-level discount amount (VND)
  line_total    INTEGER NOT NULL DEFAULT 0         -- computed by Rust, see §4
);
CREATE INDEX idx_items_order ON order_items(order_id);

CREATE TABLE order_item_toppings (
  id            INTEGER PRIMARY KEY,
  order_item_id INTEGER NOT NULL REFERENCES order_items(id) ON DELETE CASCADE,
  topping_id    INTEGER REFERENCES toppings(id),   -- nullable: topping may be deleted later
  topping_name  TEXT NOT NULL,                     -- snapshot
  price         INTEGER NOT NULL DEFAULT 0,        -- snapshot unit price
  quantity      INTEGER NOT NULL DEFAULT 1 CHECK (quantity > 0)
);
CREATE INDEX idx_item_toppings_item ON order_item_toppings(order_item_id);
```

## 2. Cross-milestone forward-reference resolution

### 2.1 `orders.shift_id` vs `shifts` (CHOSEN APPROACH — binding)

**Decision: `orders.shift_id` is a NULLABLE `INTEGER` column with NO hard SQL foreign key.** It is created in the M3 migration (`0004_orders.sql`) and populated by M5 logic. This is the recommended approach and is now mandatory for all plans.

Rationale and rules:
- M3 ships before M5, so a `REFERENCES shifts(id)` constraint in `0004` would reference a table that does not yet exist. Declaring `shift_id INTEGER` (no FK) sidesteps the ordering problem with zero forward dependency.
- M3 may write `orders` with `shift_id = NULL`. M3 plans MUST NOT assume a shift exists.
- **M5 owns the linkage.** When M5 lands: `open_shift` creates the OPEN shift; `create_order` (which lives in M3 but is *extended* in M5, or reads current shift via a shared helper) stamps `orders.shift_id` with the current open shift's id. M5 documents this stamping in its plan and adds a repo helper `current_open_shift_id(conn) -> Option<i64>`.
- Referential integrity for `shift_id` is enforced in **application code** (repo/service layer), not by SQLite. M5 tests cover "order created during open shift gets that shift_id" and "expected_cash sums CASH payments of orders in this shift".
- The spec's prose "Mọi đơn gắn `shift_id` của ca đang mở" is honored at runtime by M5; the schema keeps it nullable to allow the M3-before-M5 build order and to tolerate (defensively) an order created when no shift is open.

> Do NOT, in any plan, add `REFERENCES shifts(id)` to `orders.shift_id`. Do NOT create the `shifts` table early in M3. The single agreed location for `shifts` is `0006_shifts.sql` (M5).

### 2.2 Other forward references — audit

| Reference | Direction | Resolution |
|---|---|---|
| `tables.area_id → areas(id)` | within M2 | Hard FK, same migration. OK. |
| `orders.table_id → tables(id)` | M3 → M2 | M2 ships first. Hard FK allowed. `table_id` is **nullable** (TAKEAWAY has no table). |
| `order_items.product_id → products(id)` | M3 → M1 | M1 ships first. Hard FK allowed but **nullable** (product may be deleted later; snapshot columns preserve the bill). |
| `order_item_toppings.topping_id → toppings(id)` | M3 → M1 | M1 ships first. Hard FK, **nullable**, snapshot preserved. |
| `order_discounts.discount_id → discounts(id)` | within M4 | Hard FK, **nullable** (manual discount has no preset row). Same migration. |
| `payments.order_id → orders(id)` | M4 → M3 | M3 ships first. Hard FK + `ON DELETE CASCADE`. OK. |
| `order_discounts.order_id → orders(id)` | M4 → M3 | M3 ships first. Hard FK + cascade. OK. |
| `orders.shift_id → shifts(id)` | M3 ↔ M5 | **NO FK** (see §2.1). |

**Rule:** any FK whose target ships in an EARLIER (lower-numbered) migration may be a hard FK. Any FK whose target ships LATER must be a plain nullable column with app-level integrity. Only `orders.shift_id` falls in the latter case.

## 3. Cross-milestone command / API surface

All commands follow the M0 shape exactly:
- Rust: `#[tauri::command] pub fn name(db: State<AppDb>, ...) -> Result<T, String>`; first line `let conn = db.0.lock().map_err(|e| e.to_string())?;` (or a private `lock(&db)?` helper as in M1); call repo; `.map_err(|e| e.to_string())`.
- Commands that touch only the filesystem (backup, PDF) take `app: tauri::AppHandle` instead of/in addition to `State<AppDb>`.
- TS wrapper: `invoke<T>("snake_command", { camelCaseArgs })`. Tauri auto-maps camelCase (JS) ↔ snake_case (Rust) for argument names. Payload structs are passed as a single `payload` arg (matches M1 `create_product`).
- Every command is registered in `tauri::generate_handler![...]` in `src-tauri/src/lib.rs`.

Argument naming: scalars are individual args; composite inputs are a single `payload: <Input>` struct. Return the **full reloaded domain object** after a write (matches M1 returning `ProductWithSizes`).

## 4. Money calculation rules (single source — referenced by M3 & M4)

All amounts are integer VND. Computed once in `services/order.rs` (line/order math) and `services/payment.rs` (payment/change). Persisted into `orders.subtotal/discount_total/total`, `order_items.line_total`, `order_discounts.amount_applied`, `payments.change_due`.

1. **Topping cost per line:** `toppings_cost = Σ (topping.price × topping.quantity)` over the item's toppings.
2. **Line total:**
   `line_total = (unit_price + toppings_cost) × quantity − line_discount`
   (`unit_price` already includes the size `price_delta`, snapshotted at add-time.) Clamp `line_total = max(0, line_total)`.
3. **Subtotal:** `subtotal = Σ line_total` over all items of the order.
4. **Order-level discount resolution** (for each `order_discounts` row, applied to `subtotal`):
   - `PERCENT`: `amount = round_vnd(subtotal × value / 100)` (value is 0..100).
   - `AMOUNT`: `amount = value`.
   - Each `amount` is clamped so cumulative discount never exceeds remaining subtotal: `amount_applied = min(amount, subtotal − already_applied)`.
5. **Discount total:** `discount_total = Σ amount_applied` over `order_discounts`.
6. **Total:** `total = max(0, subtotal − discount_total)`.
7. **Rounding:** round to whole VND using `rounding_unit` setting (default `1` = round to the dong). Rounding helper `round_vnd(x)` lives in `services` and is the ONLY place rounding happens. Default `rounding_unit=1` is effectively identity since inputs are already integers; values >1 (e.g. 500) round the final `total` to the nearest unit.
8. **Cash payment & change:** for `method = CASH` with `tendered`: `change_due = max(0, tendered − total)`. `add_payment` rejects `tendered < total` only for a single full CASH payment when finalizing; partial/split payments are summed.
9. **Finalize guard:** `finalize_order` requires `Σ payments.amount ≥ total`; on success set `status = PAID`, `paid_at`, release the table.
10. **Shift reconciliation (M5):** `expected_cash = opening_cash + Σ payments.amount WHERE method='CASH'` for PAID orders whose `shift_id` = this shift; `cash_diff = closing_cash_counted − expected_cash`; `total_sales = Σ orders.total WHERE status='PAID' AND shift_id` = this shift.

Edge rules: negative inputs rejected at repo/service boundary (mirror M1 validation returning `rusqlite::Error::InvalidParameterName`). `quantity ≥ 1`. All multi-row money writes occur inside a transaction.

**Repo pattern (binding):** functions take `conn: &Connection` (or `&mut Connection` only when they internally `BEGIN`), return `rusqlite::Result<T>`, validate inputs and return `rusqlite::Error::InvalidParameterName("…")` (Vietnamese message) for bad data. Row-mapper helper `row_to_<x>` per struct. Booleans read as `row.get::<_, i64>("col")? != 0`. Reload-and-return after writes.

**Transaction pattern (binding):** any write spanning multiple tables (orders+items+toppings, apply discount, finalize, merge) uses the M1 `create_product` pattern — `conn.execute_batch("BEGIN")`, do work in an inner closure, `COMMIT` on `Ok` / `ROLLBACK` on `Err`. `foreign_keys=ON` is already set by `open_and_migrate`; `ON DELETE CASCADE` is relied on for item/topping/discount/payment children.

**TDD (binding for all Rust logic):** write the failing repo/service test first (in-memory `Connection` seeded via `migrations::run`), see it fail, implement, see it pass — exactly the M1 cadence. **Money math in `services/*` MUST be unit-tested** (spec §10.8 calls this the highest-risk area): line_total, subtotal, percent/amount discounts, clamping, change_due, shift reconciliation. Frontend gets Vitest only for non-trivial logic (cart math display, option popup); plain CRUD screens follow the M1 try/catch + error-state pattern without mandatory component tests.

**Frontend rules (binding):** React NEVER touches SQL — only `invoke` via `lib/api/*`. All async calls wrapped in `try/catch` with an `error` state (M1/Settings pattern). Money rendered with `formatVnd` from `src/lib/format.ts`; never format VND inline. Vietnamese UI strings. Touch-friendly (buttons ≥ 44px, number pad for cash).
